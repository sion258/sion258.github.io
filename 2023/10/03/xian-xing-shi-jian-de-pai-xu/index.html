线性时间排序

一. 比较排序算法的下界和决策树模型

在我们之前了解的排序算法中，都依赖于元素之间的比较，我们称这些排序方法为比较排序（comparsion sort）.

给你几分钟回忆一下我们学过的排序方法的时间复杂度为多少？

对于n个元素的输入序列：

  • 插入排序O(n^2)
  • 归并排序O(nlgn)
  • 推排序O(nlgn)
  • 快速排序O(nlgn)

其中快速排序在划分为0和n-1时的情况下退化为O(n^2),随机化快速排序为O(nlgn)，其中归并排序和堆排序在最坏的情况下仍要经过下界Omega(nlgn)次比较，因此归并排序和推排序都是渐进最优的。

那么所有的比较排序的比较次数的至少是Omega(nlgn)吗？有没有比O(nlgn)更快的排序算法呢？

我们借助决策树(Decision Tree)来进行分析。

例如，对于一个三元素序列，我们可以将其比较排序的方式抽象为：

可以观察到这棵树的叶子节点包含了所有的可能的比较情况，对于3个元素的比较，至少包含3！的可能的排序。合理外推，可以得到对于n个元素的比较，包含n！种的可能排序，即n！个叶子节点。

在树中，最坏的情况对应最多的比较次数，观察决策树，我们可以发现比较次数即到达一个叶子节点的高度，最坏的情况对应最深的叶子节点，即这个决策树的高，记作h。在一个二叉树中，叶子节点最多的情况就是这个树是一棵满二叉树，当一个棵树是满二叉树时，叶子节点为2^h,因此决策树的叶子节点l至多为2^h。那么决策树的叶子节点l满足：
$$
n!<=l<=2^h
$$
则
$$
h>=lg(n!)
$$
由斯特林公式(Stirling’s approximation)
$$
n!\approx\ \sqrt{2\pi n}(\frac{n}{e})^n
$$
可以化简得到：
$$
h = \omega(nlgn)
$$
我们可以发现比较次数至少为nlgn，所以所有的比较排序的下界Omega(nlgn)。

由于归并排序和堆排序的运行时间上界为O(nlgn),所以他们是渐进最优的比较排序方法。

那么有没有运行时间更小的排序算法呢？

二. 线性时间的排序算法

1. 计数排序

1.1 原理

想象一下，不利用数与数之间的比较，是否能实现一组输入序列的排序？

比如有一组数<2,3,4,1,5>,有5个元素，范围1~5，比5小的数有4个，那么5应该在第五个元素的位置上，4应该在第四个元素的位置上，…,1应该在第1的位置上。那么对于一个元素x，确定比x小的元素的个数，可以确定x在输出序列的位置。

所以，已知一个输入序列在0~k的范围内，通过分配k个计数器，统计输入序列中的元素i在第i个计数器上的出现次数，将计数器求前缀和，就有了每一个元素在输出序列的具体位置。

1.2 代码

知道了思想，我们可以写出相应的实现过程，这里我将写出计数排序的java实现：

public static  CountingSort(int[] a){
	int[] c = new int[max(a)];
	int[] b = new int[a.length];
	
	for(int i = 1;i <= a.length;i++){
		int j = a[i];
		c[j]++;
	}
	for(int i = 1;i <= max(a);i++){
		c[i] = c[i-1]+c[i]l;
	}
	for(int i = a.length;i>=1;i++){
		int j = a[i];
		b[c[j]]= a[i];

	}
}

下面是分步图解：

我们新立数组c来作为计数器，长度为a的最大值，因为我们要对0~k的每一个数都设立计数器统计每个数出现的次数，随后求前缀和为更新的c数组，以确定各个数在输出数组b出现的位置，之后从a.length开始遍历，每遍历一个a[i],找到统计该a[i]的计数器c[a[i]]，此时更新为前缀和的数组c代表的是其下标，也就是a[i]，在输出序列的位置，所以我们将a[i]的值赋给b[c[a[i]]]，c[a[i]]代表的a[i]的计数器减一，表示已经等于a[i]的值输出过一次，下一次遍历到与a[i]相同的值的时候会把这个值放在a[i]的前一位。

1.3 性能分析

计数排序的时间复杂度是多少呢？

第一个for循环的时间复杂度是O(n)，第二个for循环的时间复杂度为O(k)，第三个循环的时间复杂度为O(n)。因此，计数排序的时间复杂度可以用O(n+k)来表示。

它是稳定的。这是因为相同序列的值在输出序列中的相对位置和在输出序列中相同。换言之，对于两个相同的数，先出现在输入序列中的数在输出序列中也先出现。

我们看到，计数排序的时间复杂度已经低于了比较排序的时间复杂度下界Omega(nlgn),但是开辟了一个k大小的数组c，因此**空间复杂度为O(k)**。

但是，当k较小时，此算法的性能不错，但是当k，譬如k=n^2之类，此算法的性能大大下降，因此对于大数之间的比较，计数算法不合适。

因此计数排序算法常常作为一个子过程在基数排序(radix sort)中.

2. 基数排序

2.1 原理

想想如何对几个已知年月日的日期排序？

2023.9.27

2023.8.21

2010.8.23

2077.8.31

我们可以从日期开始进行排序，比如我们在这里从小到大排序：

2023.8.21

2010.8.23

2023.9.27

2077.8.31

接着对月份排序：

2023.8.21

2010.8.23

2077.8.31

2023.9.27

接着对年份排序

2010.8.23

2023.8.21

2023.9.27

2077.8.31

我们可以发现从最小位对其所有“位”排序之后，我们得到了正确的顺序1，这就是基数排序的基本思想。

从最低有效位排序，接着向高位排序，结果即是排序好的结果。

根据原理而言，我们需要选择一个稳定的排序方法作为每一位的排序方法，比如上面的排序过程，有两个为2023的年份，排序结果必须与上一位排序的相对位置相同，即2023.8在2023.9之前。

所以我们需要一个稳定的排序方法，比如对十进制数的排序，每一位数在0~9之间，范围不大，所以我们可以选择计数排序作为每一位的排序方法。

2.2 代码

public class RadixSort {

    // 获取数组中最大的数
    public static int getMax(int[] arr) {
        int max = arr[0];
        for (int i = 1; i < arr.length; i++) {
            if (arr[i] > max) {
                max = arr[i];
            }
        }
        return max;
    }

    // 使用计数排序对数组按照指定的位数进行排序
    public static void countingSort(int[] arr, int exp) {
        int n = arr.length;
        int[] output = new int[n];
        int[] count = new int[10];
        Arrays.fill(count, 0);

        // 统计每个数字的出现次数
        for (int i = 0; i < n; i++) {
            count[(arr[i] / exp) % 10]++;
        }

        // 计算每个数字应该在输出数组中的位置
        for (int i = 1; i < 10; i++) {
            count[i] += count[i - 1];
        }

        // 构建输出数组
        for (int i = n - 1; i >= 0; i--) {
            output[count[(arr[i] / exp) % 10] - 1] = arr[i];
            count[(arr[i] / exp) % 10]--;
        }

        // 将输出数组复制到原始数组中
        for (int i = 0; i < n; i++) {
            arr[i] = output[i];
        }
    }

    // 使用基数排序对数组进行排序
    public static void radixSort(int[] arr) {
        int max = getMax(arr);

        // 从最低位到最高位依次进行排序
        for (int exp = 1; max / exp > 0; exp *= 10) {
            countingSort(arr, exp);
        }
    }

2.3 性能分析

​ 如果基数排序使用的稳定的排序方式(以计数排序为例)的时间复杂度为O(n+k)，最大位数为d，那么基数排序的时间复杂度为**O(d(n+k))**。因为每一轮消耗的时间为O(n+k),有d位数，那么就进行d轮。

2.4 拓展

为了减少k的大小，我们可以将要比较的数转换为二进制，这样k的大小为1，大大减小了k的的大小和计数排序的时间复杂度，但是位数将展开得非常大，如何在二者之间取平衡呢？

假设被排序的数换算成b个bit(b位数)，我们可以将多位合成一组，假如r个bit一组，那么可以看作b/r个r位数，那么计数排序的轮数就是b/r轮，每一位数在范围0~2^r-1内，那么时间复杂度可以表示为：
$$
O((b/r)(n+2^r))
$$
由于位数b和元素个数n给定，我们是否能找到一个r使得时间复杂度最低？

如果b<=lgn,对于任何满足r<=b的r，都有n+2^r=O(n),因此当b=r时，有渐进意义最优的O((b/r)(n+2^r)) = O(n)；

如果b>lgn,r<=lgn,都有n+2^r = O(n)，当r = lgn，时间复杂度为O(bn/lgn)，接下来如果r变大，分子2^r增大的速度大于分母r增大的速度，所以这时候的时间复杂度大于O(bn/lgn)；相反r开始变小，分母变小(比b/lgn更大)，而始终有n+2^r = O(n)，时间复杂度大于O(bn/lgn)。因此b>lgn时，时间复杂度O(bn/lgn);

当然，更精确的计算结果可以通过对其求导获得。
